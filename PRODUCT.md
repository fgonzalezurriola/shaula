# Product

## Register

brand

## Platform

web

## Users

Wayland users looking for a native screenshot workflow. They want to confirm compatibility quickly, see the product work, and install it without reading a marketing page.

## Product Purpose

Shaula captures, annotates, saves, and copies screenshots on Wayland. The website should get a visitor from recognition to installation with as little friction as possible.

## Positioning

A native, local screenshot workflow for Wayland that chooses a supported capture route automatically.

## Conversion & proof

- Primary CTA: install Shaula.
- Secondary CTA: inspect or star the project on GitHub.
- The line a visitor remembers after 10 seconds: Screenshots for Wayland.
- Belief ladder: Shaula is for Wayland; installation is straightforward; the real workflow works; the project is open source.
- Proof on hand: the real capture and annotation demo at `web/public/shaula-demo.gif`, public source, and release artifacts.

## Brand Personality

Direct, capable, restrained. The page should feel calm and technically credible without dressing plain facts as marketing copy.

## Anti-references

Avoid generic SaaS copy, feature inventories, repeated section kickers, decorative metrics, terminal cosplay, glow effects, gradient treatments, glassmorphism, and any text that sounds like a pitch deck.

## Design Principles

- Say what Shaula is, then show it.
- Keep installation closer than explanation.
- Let the real product demo provide the proof.
- Use technical detail only when it helps someone act.
- End with the open-source project, not another sales pitch.

## Accessibility & Inclusion

Meet WCAG AA contrast, preserve visible keyboard focus, provide 44px touch targets, announce clipboard results, and respect reduced-motion preferences.
